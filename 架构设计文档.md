## 一、关于「是否生产级」

**是的，目标就是生产级。** 但生产级 ≠ 一次性交付所有层。策略是：分层设计、分期交付。

- Phase 1 跑通核心循环（Planner → Executor → Reflector），用最少层证明价值
- Phase 2+ 补齐 Gateway / Memory / 观测 / 安全，达到可多用户部署
- 每层有清晰接口契约，不因分期而推倒重来

## 二、框架适配层（最高优先级）

### 2.1 问题定义

市面上主流 AI 编码框架各有不同的协议：

| 框架 | 协议风格 | 工具调用 | 记忆模型 | 安全策略 |
|------|---------|:---:|:---:|:---:|
| **Claude Code** | Anthropic 官方 CLI，MCP 协议 | OAuth + Tool Use | Session-based | 内置 |
| **Codex (OpenAI)** | OpenAI API，Function Calling | 原生 Function Call | 无内置持久 | API Key |
| **Hermes** | 开源自主 Agent 框架 | 插件系统 | 向量数据库 | 自定义 |
| **Coze** | 字节跳动 Agent 平台 | 插件市场 | 内置变量系统 | 平台级 |
| **Marvis** | 腾讯桌面 Agent | dispatch_task | memory_ids | 内置 |

### 2.2 解决方案：统一接口契约 + 适配器模式

```
                     ┌─────────────────────────┐
                     │     Agent 接口契约        │
                     │  (Protocol Interface)    │
                     │                          │
                     │  - task: string          │
                     │  - context: Context      │
                     │  - tools: ToolSchema[]   │
                     │  - stream: bool          │
                     │  → result: AgentResult   │
                     └──────────┬──────────────┘
                                │
        ┌───────────┬───────────┼───────────┬───────────┐
        ▼           ▼           ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ Claude  │ │ Codex   │ │ Hermes  │ │ Coze    │ │ Marvis  │
   │ Adapter │ │ Adapter │ │ Adapter │ │ Adapter │ │ Adapter │
   └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘
```

**契约定义示例**（`agent-protocol.ts`）：

```typescript
interface AgentProtocol {
  /** 提交任务 */
  execute(request: AgentRequest): Promise<AgentResponse>;
  /** 流式执行 */
  executeStream(request: AgentRequest): AsyncIterable<AgentEvent>;
  /** 获取可用工具列表 */
  listTools(): ToolSchema[];
  /** 健康检查 */
  health(): Promise<HealthStatus>;
}

interface AgentRequest {
  task: string;                    // 当前任务描述
  overallGoal?: string;            // 总目标（多步骤时）
  context: Context;                // 上下文（文件/历史/状态）
  tools: ToolSchema[];             // 可用工具
  options?: { maxSteps: number; temperature: number };
}

interface AgentResponse {
  status: 'success' | 'partial' | 'failed';
  result: unknown;                 // 结构化产出物
  steps: StepRecord[];             // 执行步骤记录
  artifacts: Artifact[];           // 产出物清单
  errors?: AgentError[];
}
```

**关键原则**：
- 每个适配器只做协议转换，不做业务逻辑
- 新框架接入 = 实现一个适配器，零侵入现有代码
- 适配器自带集成测试，验证与目标框架的兼容性

## 四、各层职责详解

### 4.1 API Gateway 层

| 职责 | 实现方式 | Phase |
|------|---------|:---:|
| 鉴权 | JWT + API Key 双模式 | Phase 2 |
| 限流 | Token Bucket，每用户每秒 N 请求 | Phase 2 |
| 路由 | 按 task type 路由到对应 Agent 实例 | Phase 1 |
| 参数校验 | Zod Schema 校验入参 | Phase 1 |
| 请求日志 | 结构化日志（不记录敏感字段） | Phase 1 |

### 4.2 编排引擎层（核心）

这是 Agent 的「大脑」，由三个组件构成闭环：

#### Planner（任务拆解）

**输入**：用户原始需求 + 上下文（项目状态、引擎选择、已有产物）

**输出**：结构化的执行计划，每个步骤包含 5 个精细化产物：

```typescript
interface PlanStep {
  id: string;
  phase: '立项' | '原型' | '生产' | '测试' | '发行' | '运营';

  // 5 个精细化产物
  directoryStructure: string[];     // 本步骤创建的目录
  interfaceContracts: Contract[];   // 接口/函数签名
  dataModels: DataModel[];          // 数据结构定义
  acceptanceCriteria: string[];     // 验收标准（可自动验证）
  estimatedTools: string[];         // 预计使用的工具

  dependencies: string[];           // 依赖的前置步骤 ID
}
```

**Planner 输出示例**（玩家背包系统）：

```
Step: implement-inventory
├── 目录结构: src/game/inventory/
├── 接口契约:
│   ├── InventoryManager.addItem(item: Item): boolean
│   ├── InventoryManager.removeItem(id: string): void
│   └── InventoryManager.getItems(): Item[]
├── 数据模型:
│   └── Item { id, name, type, quantity, maxStack }
├── 验收标准:
│   ├── [ ] 添加道具后 getItems() 返回新增项
│   ├── [ ] 堆叠道具不超过 maxStack
│   └── [ ] 移除不存在的道具时返回 false
└── 预计工具: write_file, python_executor(test)
```

#### Executor（按步骤执行）

- 严格按照 Planner 生成的步骤顺序执行
- 每个步骤调用对应工具，记录工具返回
- 步骤间有依赖时等待前置步骤完成
- 失败时记录到 StepRecord，不回滚已成功的步骤（除非用户要求）

#### Reflector（反思校验）

Executor 完成每个步骤后，Reflector 自动介入：

| 校验维度 | 检查内容 | 失败处理 |
|---------|---------|---------|
| **文件存在性** | 目录结构是否创建 | 补充创建 |
| **接口一致性** | 函数签名是否匹配契约 | 自动修正或标记 |
| **数据模型** | 类型定义是否符合规范 | 补充缺失字段 |
| **验收标准** | 逐条跑测试 | 未通过则回到 Executor |
| **代码质量** | Lint / 类型检查 | 自动修复或标记 |

**Reflector 循环上限**：同一步骤最多反射 3 轮，超限则降级为人工介入。

### 4.3 记忆层

| 类型 | 存储 | 生命周期 | 示例 |
|------|------|---------|------|
| **会话上下文** | 内存 + Redis | 单次会话 | 当前步骤、已执行历史 |
| **项目记忆** | 文件系统 | 项目周期 | 技术选型、目录结构、引擎配置 |
| **用户偏好** | SQLite | 永久 | 代码风格、语言、引擎偏好 |
| **知识库** | 向量数据库 | 永久 | 引擎文档、设计模式、常见问题 |

### 4.4 安全过滤层

| 位置 | 机制 |
|------|------|
| **输入净化** | 注入检测（prompt injection）、长度限制、编码校验 |
| **输出校验** | 危险代码检测（rm -rf、eval、exec）、路径遍历防护 |
| **敏感词过滤** | 不合规内容的拦截（如涉政/涉黄） |

### 4.5 观测层

| 维度 | 工具 | 指标 |
|------|------|------|
| 链路追踪 | OpenTelemetry | Planner→Executor→Reflector 全链路 |
| 指标采集 | Prometheus | 步骤耗时、成功率、反思轮次 |
| 审计日志 | 结构化文件 | 谁、何时、做了什么、产出什么 |
| 告警 | — | Reflector 3 轮失败 / Executor 异常 |

## 五、前端交互层设计要点

> 聚焦：**状态 | 交互 | 边界态**

### 5.1 状态管理

```
┌─────────────────────────────────────────┐
│              Agent 会话状态机             │
│                                          │
│  idle → planning → executing → reflecting │
│    ↑        │          │           │      │
│    └────────┴──────────┴───────────┘      │
│          (可中断/可恢复)                   │
└─────────────────────────────────────────┘
```

每个状态对应不同的 UI 渲染：

| 状态 | UI 表现 |
|------|--------|
| `idle` | 输入框可用，展示上次结果 |
| `planning` | 骨架屏 + "规划中，正在拆解任务..." |
| `executing` | 步骤进度条 + 实时日志流 |
| `reflecting` | 每步校验结果 + 修正提示 |
| `error` | 错误信息 + 重试/跳过/人工介入 |

### 5.2 交互

- **流式输出**：Executor 每步结果通过 WebSocket 实时推送
- **可中断**：用户随时暂停/取消当前 Planner 计划
- **人工介入**：Reflector 3 轮失败后弹出决策面板（重试/跳过/手动修复）
- **产物预览**：每个步骤完成后渲染产物卡片（文件/图片/数据表）

### 5.3 边界态

| 场景 | 处理 |
|------|------|
| 网络断开 | 显示断线状态，WebSocket 自动重连 |
| 会话过期 | 提示重新登录，保留未提交输入 |
| 工具超时 | 单工具 30s 超时，整步骤 300s 超时 |
| 长任务 | 显示预估剩余时间，支持后台执行 |
| 空状态 | 首次使用引导、无历史记录时展示示例 |
| 错误恢复 | 自动保存草稿，刷新后恢复 |

## 六、后端工程规范

| 维度 | 规范 |
|------|------|
| **鉴权** | 所有 API 需 JWT；工具执行需二次校验权限范围 |
| **参数校验** | Zod Schema，拒绝不符合 Schema 的请求 |
| **异常处理** | 统一 Error Code 枚举，不暴露堆栈到前端 |
| **日志** | 结构化日志（JSON），分 INFO/WARN/ERROR/DEBUG |
| **测试** | 单元（工具函数）+ 集成（适配器）+ E2E（完整管线） |

## 七、项目开发纪律（兜底）

| 纪律 | 工具/方式 | 触发时机 |
|------|---------|---------|
| **小步生成** | 每个步骤 ≤ 200 行代码 | Planner 约束 |
| **自动化测试** | Vitest（前端）/ Pytest（后端） | 每步 Executor 后 |
| **Lint 检查** | ESLint + Prettier + tsc --noEmit | 每步 Reflector |
| **代码 Review** | Reflector 自动 Review + 产物校验 | 每步 Reflector |
| **依赖白名单** | 仅使用 package.json 中声明的依赖 | Executor 工具调用前检查 |
| **产物校验** | 对照 Planner 的 acceptanceCriteria | Reflector |
